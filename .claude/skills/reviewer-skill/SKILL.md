# 审查技能包

[技能说明]
    章节审查技能，用于在章节写作完成后进行质量检查。审查分步骤执行，默认文章有问题，任务是找出来。问题按两类归组输出：结构/逻辑类（writer处理）和表达类（polisher处理），并标注问题来源（骨架层/扩展层/表达层）

[文件结构]
    reviewer-skill/
    ├── SKILL.md                  # 本文件
    ├── review-checklist.md       # 审查清单与评分维度
    ├── quality-standards.md      # 质量标准定义
    └── templates/
        └── review-report-template.md  # 审查报告模板

    共享技能（审查时参考）：
    - shared/deai-rules.md
    - shared/hook-techniques.md

[功能]
    [审查章节]
        第一步：读取输入
            - 待审查的章节正文（chapters/chNN.md）
            - 该章的场景规划（plans/chNN-plan.md，来自planner）
            - 骨架稿（plans/chNN-skeleton.md，如存在）
            - 前一章的场景规划（plans/ch(N-1)-plan.md，用于钩子链验证）
            - 该章涉及的人物档案当前状态字段
            - 前章结尾约500字（用于连贯性审查）
            - bible/plot/suspense-tracker.md（用于悬念维度）
            - bible/plot/foreshadow-tracker.md（用于伏笔维度）
            - bible/plot/decisions.md（用于理解规划中的决策背景和偏离理由）
            - bible/plot/outline.md（读取体裁字段用于权重调整）

        第二步：读取审查标准
            - 读取 review-checklist.md
            - 读取 quality-standards.md
            - 执行审查基准建立，参考review-checklist.md
            - 根据 outline.md 中的体裁字段，按 review-checklist.md 中的体裁权重调整审查侧重

        第三步：按需读取参考文件
            - 检查AI味时参考 shared/deai-rules.md
            - 检查钩子和悬念时参考 shared/hook-techniques.md

        第四步：读取模板
            - 读取 templates/review-report-template.md

        第五步：逐维度审查
            按 review-checklist.md 中定义的维度逐项检查，
            特别注意：
            - 维度四：承重 beat 是否在场、顺序是否完整、是否有大纲语言残留
            - 维度五：情感弧是否达成（入口X → 出口Y）、节奏权重是否匹配
            - 维度九：钩子链是否连续（上章出口是否被回应、本章出口是否可信）
            - 骨架忠实度：正文与骨架的结构偏差是否合理
            每个维度给出：
            - 评分（1-5分）
            - 问题描述（如有）
            - 具体位置（引用原文）
            - 修改建议

        第六步：判断处理方
            对每个问题判断应由谁处理：
            结构/逻辑类（writer处理）：
            - 叙事逻辑问题
            - 人物一致性问题（行为与设定矛盾）
            - 场景规划执行问题（遗漏核心事件、承重 beat 缺失）
            - 连贯性问题（时间线、位置矛盾）
            - 节奏结构问题（需要重写场景）
            - 内容缺失问题（场景被跳过或严重压缩）
            - 钩子链断裂（上章钩子未被回应）
            - 大纲语言残留

            表达类（polisher处理）：
            - AI味问题
            - 对话语气措辞
            - 描写精炼
            - 节奏微调（不涉及结构）
            - 措辞优化
            每个问题同时标注来源：
            [骨架层] / [扩展层] / [表达层]

        第七步：就绪判定
            按 review-checklist.md 中的评分汇总标准生成就绪判定：
            Green：总分≥35，无单项≤2，无红线触发，骨架忠实执行
            Yellow：总分25-34，或有一项=2，或轻微节奏违规，或骨架部分偏离
            Red：总分<25，或核心维度≤1，或红线触发

            就绪判定包含：
            - 判定结果（Green/Yellow/Red）
            - 判定信号列表（哪些检查项导致此判定）
            - 红牌列表（如有触发红线）
            - 阻塞项（Red 时列出必须修复的问题）

        第八步：生成报告
            按模板格式输出审查报告，包括：
            - 总评分
            - 各维度评分和问题
            - 修改建议清单（标注处理方和问题来源）
            - 钩子链验证结果
            - 骨架忠实度评估
            - 就绪判定结果

        第九步：写入文件
            - 写入 reviews/chNN-review.md
            - 返回完成确认

    [复审章节]
        当章节经过修改后再次提交审查时触发：
        第一步：读取修改后的章节和上一份审查报告
        第二步：检查上一份报告中的问题是否已解决
        第三步：检查修改是否引入了新问题
        第四步：生成复审报告
        第五步：覆写 reviews/chNN-review.md，在文件头标注版本号 [复审版本 vN]

[审查结论标准]
    通过：总分≥35分（满分50分），无单项≤2分
    需修改：总分25-34分，或有单项≤2分但整体可救
    需重写：总分<25分，或核心维度（叙事逻辑/人物一致性）≤1分

[注意事项]
    - reviewer只输出报告，不直接修改章节
    - 默认文章有问题，没有发现问题不代表文章好，只代表审查不够仔细
    - 问题描述要具体，引用原文位置，不说"整体感觉不好"
    - 修改建议要可执行，不说"写得更好一点"
    - 不修改 bible/ 目录下任何文件
